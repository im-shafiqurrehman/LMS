# Enrollment and Payments — Set the Scene

Money changes things. When a student pays for a course, you have a financial record. You must not double-charge. You must not partially fulfill the transaction. If the payment succeeds but the enrollment fails, the student paid but cannot access the course. These are not edge cases — they are the normal failure modes of distributed systems.

This chapter builds the payment and enrollment flow:
1. Student requests a Stripe payment intent (a "pending payment record")
2. Student completes payment in the browser (handled by Stripe's frontend SDK — chapter 14)
3. Stripe sends a webhook to the server confirming payment
4. Server creates the Enrollment — with idempotency to prevent duplicates

The key safety mechanism: Stripe's stripePaymentIntentId is stored on every Enrollment document. A duplicate webhook for the same payment intent is silently ignored.
