# Idempotency — Why It Matters

Idempotency means calling the same operation multiple times has the same effect as calling it once. For payment processing, this is not a nice-to-have — it is required.

Stripe's webhook system guarantees "at least once" delivery. In a network failure, the same payment_intent.succeeded event may arrive twice, five minutes apart. Without idempotency:

- First delivery: Enrollment created ✅
- Second delivery: Another Enrollment created ❌ — student enrolled twice, double-counted revenue

With idempotency:
- First delivery: Enrollment created ✅
- Second delivery: findOne finds the existing enrollment, returns immediately ✅

The unique index on stripePaymentIntentId provides a database-level fallback: even if the application logic has a race condition, the database will reject the second Enrollment.create() with a duplicate key error.
