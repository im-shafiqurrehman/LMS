# Offset vs Cursor Pagination

**Offset pagination** (the most common implementation):

```js
const courses = await Course.find(filter)
  .skip(page * limit)
  .limit(limit);
```

Problems with offset at scale:
- skip(n) requires MongoDB to read and discard n documents before returning results
- At page 100 with limit 12, MongoDB reads 1200 documents and throws away 1188
- If a new course is added while a user is on page 3, they may see a duplicate or miss a course on page 4 (the "page drift" problem)

**Cursor pagination** (what EduFlow uses):

```js
const filter = { status: 'published' };
if (cursor) filter._id = { $lt: cursor }; // "give me items before this ID"
const courses = await Course.find(filter)
  .sort({ _id: -1 })  // newest first, using the built-in _id index
  .limit(limit + 1);  // fetch one extra to detect if there are more pages

const hasMore = courses.length > limit;
const items   = courses.slice(0, limit);
const nextCursor = hasMore ? items[items.length - 1]._id : null;
```

The client passes the last item's _id as the cursor for the next page. MongoDB uses the _id index directly — no scan, no drift.

✅ Use cursor pagination for all list endpoints in EduFlow.
❌ Do not use skip() for production list endpoints.
