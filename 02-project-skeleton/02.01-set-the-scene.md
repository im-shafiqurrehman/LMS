# Project Skeleton — Set the Scene

> "Before a single feature works, the scaffold must stand."

Every professional developer has a story about the project where someone skipped the setup phase — copied a folder from an old project, patched a config file, and pushed. Three weeks later, secrets ended up in Git, the frontend and backend disagreed on what the API URL was, and the production deployment was a surprise because nobody had ever done it before.

EduFlow starts differently. Before you write a register endpoint or a course catalogue, both applications will exist in a clean repository, both will boot, and the frontend will talk to the backend through a health check. This is the "project skeleton" — the structural commitments you make once and never revisit.

---

## What this chapter covers

By the end of this chapter, EduFlow has two running applications in one repository:

- **server/** — an Express.js backend that listens on port 3000 and responds to GET /api/health
- **client/** — a Next.js frontend that runs on port 3001 and fetches from the backend

Both apps boot cleanly, read configuration from .env files, and have their secrets excluded from Git. The Mongoose connection is wired but not yet tested against a database — that comes in chapter 04. For now, the skeleton stands.
