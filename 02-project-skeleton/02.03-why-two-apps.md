# Why Two Separate Apps?

A natural first instinct is to build EduFlow as a single application — put the API routes and the React components in one Next.js project. Next.js supports API routes; why not use them?

**The problem with API routes in Next.js for a production LMS:**

1. **Deployment coupling.** If you want to scale the API independently from the UI (more API servers during an enrollment burst), you cannot — they are the same deployed unit.
2. **Framework lock-in.** A standalone Express API can be consumed by the Next.js frontend, a mobile app, and a future admin CLI. A Next.js API route cannot.
3. **Configuration mismatch.** Next.js is optimised for React rendering. Express is optimised for stateless HTTP APIs. Trying to configure Mongoose connection pooling, process managers, and background jobs inside a Next.js deployment is painful.

**The rule:** Express owns the data and business logic. Next.js owns the presentation. They communicate over HTTP — the same way any other client would.

This separation is also the industry standard. Understanding how a frontend-backend pair works prepares you for every real project you will join.

✅ Backend in server/, frontend in client/, both in one monorepo root.
❌ Do not add API routes to the Next.js app. All data lives in Express.
