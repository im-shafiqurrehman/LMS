# Mongoose vs the Native MongoDB Driver

MongoDB's native Node.js driver is the lowest-level way to talk to MongoDB. It gives you direct access to collections:

```js
// Native driver — shape only
const db = client.db('eduflow');
const users = db.collection('users');
const user = await users.findOne({ email: 'alice@example.com' });
```

This works — but it lacks three things that matter for a production application:

**1. Schema validation.** The native driver accepts any document shape. A typo in a field name silently creates a new field in the database. Mongoose puts a schema layer in application code and validates every document before writing it.

**2. Middleware hooks.** Mongoose lets you run code automatically before or after operations. The most critical example: a pre('save') hook that hashes a user's password before every write. With the native driver, you must remember to call bcrypt.hash() in every route handler that touches the password. Forget once, and you have stored a plain-text password.

**3. Populate (reference resolution).** MongoDB stores references as ObjectIds. The native driver requires separate queries to resolve them. Mongoose's .populate() resolves references in a single readable line:

```js
const course = await Course.findById(id).populate('instructorId', 'name email');
```

Behind the scenes, Mongoose issues the lookup efficiently. You write one line.

**When to use the native driver instead of Mongoose:** for large aggregation pipelines, bulk write operations, and change streams where Mongoose's abstraction adds friction. For EduFlow's query patterns, Mongoose is the right tool throughout.
