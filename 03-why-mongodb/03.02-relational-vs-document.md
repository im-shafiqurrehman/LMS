# Relational vs Document Databases

**Relational databases** (PostgreSQL, MySQL) store data in tables — rows and columns. Relationships between tables are expressed as foreign keys: a Lesson row has a courseId column that references the Course table's primary key. The database engine enforces these relationships; you cannot insert a Lesson with a courseId that does not exist. The structure is rigid but trustworthy — the database rejects invalid data.

**Document databases** (MongoDB) store data as JSON-like documents — nested objects and arrays. A Course document can contain an embedded array of category tags, a map of metadata, or any structure you choose. There is no foreign key enforcement at the database level; you declare relationships in application code.

**The trade-off in plain terms:**

| | Relational (PostgreSQL) | Document (MongoDB) |
|--|------------------------|-------------------|
| Schema | Enforced by the database | Enforced by the application (Mongoose) |
| Relationships | Foreign keys, JOIN queries | References (ObjectId), application-level joins |
| Migrations | Required for schema changes | Schema changes are add-a-field simple |
| Transactions | ACID, multi-row | ACID per document; multi-document since v4 |
| Scale | Vertical (bigger server) | Horizontal (more servers, sharding) |
| Best fit | Financial records, inventory, strong consistency | Flexible schemas, high write volume, evolving models |

Neither is universally better. The right choice depends on your data and your requirements.
