# Referencing vs Embedding in MongoDB

In a relational database, all data is in separate tables and you JOIN them. In MongoDB, you have a choice for each relationship: **reference** the other document (store its ObjectId and look it up later) or **embed** the related data directly inside the parent document.

**When to embed:**
- The embedded data is small and always needed with its parent
- The embedded data rarely changes independently
- The embedded data does not need to be queried on its own

Example: a course's lesson list. If you always fetch lessons when you fetch a course, and lessons have no existence outside their course, embedding them as an array is reasonable for small course sizes.

**When to reference:**
- The related document is large or has many fields
- The related document is queried independently
- The related document is referenced by many other documents

Example: the User document. A User is referenced by Enrollments, Courses (as instructorId), ProgressRecords, Questions, and Answers. Embedding the user inside every one of these would duplicate data and make updates (a user changes their email) require updating many documents simultaneously.

**EduFlow's approach:** reference everything. The data model uses ObjectId references throughout, and Mongoose's .populate() resolves them when needed. This is the correct default for an application whose data is queried from many angles.

```
Enrollment.studentId → User._id
Enrollment.courseId  → Course._id
ProgressRecord.lessonId  → Lesson._id
Certificate.enrollmentId → Enrollment._id
```

The price: an extra query to resolve each reference. The benefit: one copy of each document, updated once.
