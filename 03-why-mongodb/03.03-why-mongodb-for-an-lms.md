# Why MongoDB for EduFlow

EduFlow has three characteristics that make MongoDB the right choice:

**1. Schema evolution.** An LMS grows — you discover new fields as you build (course tags, video metadata, referral codes, discussion threads). MongoDB lets you add fields to documents without a migration step. In PostgreSQL, adding a column to a table with 100,000 rows requires an ALTER TABLE — which locks the table and takes time. In MongoDB, you add the field in the schema definition and it appears on new documents immediately.

**2. Document-shaped data.** An LMS course is naturally a rich document: a title, a description, a cover image, a list of lessons (each with a video URL and duration), and an instructor profile. Storing this in MongoDB maps almost directly to how you think about the data. In a relational database, you would need at least four tables and a JOIN query to reconstitute a single course page.

**3. Horizontal scalability.** MongoDB's sharding model is designed for horizontal scale — distributing data across multiple servers based on a shard key. When EduFlow grows to millions of users, sharding by userId or courseId is a natural fit. PostgreSQL can scale horizontally too (with read replicas and connection pooling), but it requires more infrastructure work.

**The honest trade-off:** by choosing MongoDB, you give up referential integrity at the database level. Nothing stops a bug from writing an Enrollment with a courseId that does not exist. This is why Mongoose matters — it gives you the schema validation and relationship checking that MongoDB itself does not provide.
