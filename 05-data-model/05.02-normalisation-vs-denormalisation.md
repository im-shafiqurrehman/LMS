# Normalisation vs Denormalisation

In a normalised data model, every fact is stored once. In a denormalised model, data is duplicated for read performance.

**Normalised example (EduFlow's approach):**

The course price is stored once, on the Course document. When displaying the course catalogue, every card shows course.price. If the instructor updates the price, it changes in one place.

**Denormalised example (what EduFlow avoids for course price):**

The Enrollment document stores a copy of course.title and course.price. When a user views their enrollment history, no JOIN is needed — the data is already there.

**The catch:** if the course title changes later, every Enrollment document still shows the old title. You either accept stale data or write code to update hundreds of Enrollment documents.

**EduFlow's rule:** normalise by default. Store each fact once. Denormalise only when you have measured a performance problem — and document why.

**The one deliberate denormalisation:** amountPaid on Enrollment. This is not a cache of course.price — it is a fact about the transaction itself. At enrollment time, the student paid a specific amount. If the instructor raises the price next month, old enrollments should not silently reflect the new price. amountPaid is a snapshot, not a copy. These are fundamentally different.

> Worth reading: search "data normalisation vs denormalisation MongoDB" — the MongoDB blog has a detailed post on when each approach is appropriate.
