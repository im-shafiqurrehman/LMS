# The IDOR Threat — Insecure Direct Object Reference

An IDOR vulnerability occurs when a user can access or modify a resource by guessing or changing its ID, without the application verifying ownership.

**EduFlow example:**

```
DELETE /api/courses/abc123
Authorization: Bearer <Instructor B's token>
```

If the courses.service.js just does Course.findByIdAndDelete('abc123'), Instructor B can delete Instructor A's course by changing the ID in the request URL. The authenticate middleware confirmed the user is logged in. The authorize('instructor') middleware confirmed they are an instructor. But neither checked *whose* course it is.

The fix is ownership verification in the service layer:

```js
async function deleteCourse(courseId, userId) {
  const course = await Course.findById(courseId);
  if (!course) throw new AppError('Course not found', 404);
  if (course.instructorId.toString() !== userId) {
    throw new AppError('Forbidden', 403);
  }
  await course.deleteOne();
}
```

Every service function that modifies a resource must:
1. Find the resource
2. Check ownership (or admin bypass)
3. Perform the operation

This is not optional. IDOR is in the OWASP Top 10 every year.
