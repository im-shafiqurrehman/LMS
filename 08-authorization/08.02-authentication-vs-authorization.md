# Authentication vs Authorization

The distinction matters because they fail differently and are fixed differently.

**Authentication failure:** the user's identity cannot be verified — wrong password, expired token, no token. Fix: the user logs in again. HTTP 401.

**Authorization failure:** the user's identity is verified, but they do not have permission to do what they are trying to do — a student accessing an instructor-only route. Fix: deny access. HTTP 403.

In practice:
- `authenticate` middleware runs first, attaches req.user, returns 401 if it fails
- `authorize('instructor')` runs second, checks req.user.role, returns 403 if it fails

Never return 401 for an authenticated user being denied access — it implies they need to log in again, which they do not. 403 is the correct status.

The Express middleware chain enforces this order:

```js
router.post('/courses', authenticate, authorize('instructor'), createCourse);
```

The route is only reachable by a verified instructor. Any other combination returns the appropriate error before createCourse runs.
