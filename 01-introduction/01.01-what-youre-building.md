# What You're Building

You are going to build **EduFlow** — a Learning Management System where instructors publish courses and students enroll, watch video lessons, track their own progress, and earn certificates. It is the kind of product you see powering real online education: think Udemy or Coursera, distilled to the core and built by you, end to end.

This is not a toy. By the time you finish, you will have shipped a real, deployed application with:

- A full authentication system with email verification, access tokens, and refresh tokens
- Role-based authorization separating students, instructors, and admins
- A course catalogue with rich filtering and search
- Video lesson delivery protected behind enrollment checks — no free rides
- Per-student progress tracking and certificate generation
- An admin dashboard for platform oversight
- A production deploy with HTTPS, environment-managed secrets, and a real domain

The stack is **Node.js + Express** on the backend, **Next.js + React** on the frontend, and **MongoDB** as the document database. You will manage the database with **Mongoose** (the ODM), authenticate users with **JWT** (access + refresh tokens), store videos and files with **Cloudinary**, cache hot reads with **Redis**, and send emails via **Resend**. Docker handles local services so your machine stays clean. The whole thing deploys to a VPS (a small cloud server) behind **Nginx**.

Every technology in that list is chosen for a reason, and the reason is explained when the technology arrives — not assumed.

## Why this project builds a real career

An LMS is a system that touches nearly every concern a production backend engineer faces:

- **Data modelling with real relationships** — users, roles, courses, lessons, enrollments, progress records, certificates. The shape of your schema determines how fast queries run and how easy features are to add.
- **Authorization that actually isolates data** — an instructor must see only their own courses; a student must access only the lessons inside courses they paid for. Getting this wrong means data leaks, not just bugs.
- **Media delivery on a budget** — video files are large, and serving them directly from your database would cost you in egress fees and latency. Learning to hand that off to a specialist service (Cloudinary) is a skill every backend engineer needs.
- **Performance under realistic load** — a catalogue page that issues one database query per course is fine at ten courses and a disaster at ten thousand. You will learn to see that problem coming and fix it before it lands in production.
- **A deploy that does not embarrass you** — secrets in environment variables, HTTPS everywhere, a process manager that restarts on crash. This is table stakes for any job.

By the end, you will have a real URL you can show an interviewer, a codebase you can explain line by line, and a learning log that proves you understood every decision you made.
