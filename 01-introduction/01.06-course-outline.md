# Course Outline

EduFlow is built across **five phases**, each a week of full-time work (~50–60 hours). The phases group chapters that belong together — by the end of each phase, the project is in a coherent, demoable state.

---

## Phase 1 — Foundations (Week 1)

Get the project standing, safe, and correctly shaped before building any features.

| # | Chapter | Done when… |
|---|---------|------------|
| 1 | Introduction | You understand the product, the people, and how the course gates progress |
| 2 | Why MongoDB and Mongoose? | You can explain why a document database with Mongoose fits this project and when you would choose relational instead |
| 3 | Project setup and config | Server boots from validated env config; no secret committed to Git; MongoDB is running in Docker |
| 4 | Next.js Frontend Setup | Frontend dev server runs on a different port; `apiClient` module is ready for use |

---

## Phase 2 — Identity (Week 1–2)

Build the system that knows who you are and what you are allowed to do.

| # | Chapter | Done when… |
|---|---------|------------|
| 5 | Authentication — Register, Login, Tokens | Register and login work; tokens are issued; refresh works; email is verified before full access |
| 6 | Authorization — Roles and Data Isolation | Protected routes enforce roles; an instructor cannot touch another instructor's data; a student cannot access unenrolled lessons |

---

## Phase 3 — The Learning Product (Week 2–3)

Build the core product loop: courses, enrollment, lessons, progress, certificates.

| # | Chapter | Done when… |
|---|---------|------------|
| 7 | Course Catalogue — Browsing and Filtering | Any visitor can browse courses with category filter and price filter; results are paginated |
| 8 | Instructor Course Management | An instructor can create, edit, and delete their own courses; publish triggers admin review |
| 9 | Enrollment and Payments | A student can pay for a course via Stripe and be enrolled on webhook confirmation |
| 10 | Lesson Access and Video Delivery | An enrolled student can access lesson content; an unenrolled student gets 403; video is served via Cloudinary signed URL |
| 11 | Progress Tracking and Certificates | Progress is recorded per lesson; a certificate PDF is generated and stored when all lessons are complete |

---

## Phase 4 — Performance and Admin (Week 3–4)

Make the platform fast, searchable, and operable.

| # | Chapter | Done when… |
|---|---------|------------|
| 12 | Search | Full-text search on course titles and descriptions returns ranked results |
| 13 | Caching and Performance | The catalogue page reads from Redis; cache is invalidated when a course is published or updated; N+1 queries are gone |
| 14 | Notifications | Welcome email, enrollment confirmation, and "certificate ready" email are sent via Resend |
| 15 | Admin Dashboard | An admin can approve or reject courses; view platform-wide metrics; deactivate user accounts |
| 16 | Rate Limiting | Auth endpoints (register, login, forgot-password) reject requests above the threshold; the error is informative |

---

## Phase 5 — Ship It (Week 4)

| # | Chapter | Done when… |
|---|---------|------------|
| 17 | Deploy | A real URL, HTTPS, secrets in env, a process manager, Nginx reverse proxy |
| 18 | Document, demonstrate, and close | ERD, architecture diagram, demo video, portfolio write-up, bar-raiser |

---

## Working rhythm

Commit every day, even if the commit is small. Secrets stay out of Git from day one — there is no "I will fix it later" with a leaked key. Read the error before asking for help. Write your learning log as you go, not the night before the viva.

You will hear these again, lightly, as the course proceeds.
