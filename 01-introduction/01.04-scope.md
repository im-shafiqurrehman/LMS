# Scope

Knowing what is **out** is as important as knowing what is **in**. Every item below the line is a real feature; it is excluded because building it first would delay the features that matter more, or introduce complexity that obscures the fundamentals.

## In scope

- User registration and login with email verification
- Role system: student, instructor, admin
- Instructor course management (create, edit, publish, delete)
- Admin course approval workflow
- Student browsing the course catalogue with category and price filters
- Paid enrollment (Stripe for payment; the integration is guided, not skipped)
- Lesson access gated behind enrollment
- Per-lesson progress tracking
- Course completion certificates (generated as PDFs, stored in Cloudinary)
- Instructor dashboard: enrolled student counts and lesson completion rates
- Admin dashboard: platform-wide metrics
- Notification emails: welcome, enrollment confirmation, certificate ready
- Video upload and delivery via Cloudinary
- Full-text search on course titles and descriptions
- Redis caching for the catalogue page
- Rate limiting on auth endpoints
- A deployed, HTTPS production environment

## Out of scope

- **Live classes or webinars** — synchronous features introduce WebSocket or WebRTC complexity that would double the project scope.
- **A subscription/membership model** — the payment system is per-course enrollment. Subscriptions are a second payment product and change the auth layer significantly.
- **Comments and discussion threads** — in-lesson comments require moderation, notifications, and real-time updates. Out of scope; not out of reach later.
- **Course bundles or discount codes** — pricing logic beyond a flat per-course price.
- **Mobile native apps** — the frontend is a responsive web app. A native app is a separate project.
- **Analytics beyond the basics** — cohort analysis, funnel visualization, A/B testing. The admin dashboard shows the metrics that matter most; the rest is a data warehouse problem.
- **Automated subtitle generation** — Cloudinary supports it; integrating it cleanly requires async processing that lives at the edge of this course's scope. Marked as a bar-raiser stretch item.

## A note on payments

Stripe is included in scope because skipping it would leave enrollment as a fiction. The payment integration is guided carefully — you will not be left alone to "figure out Stripe." But the LMS is not a payments course, so the integration is kept focused: card payments, enrollment on success, webhook handling for failed payments. No refunds, no disputes, no invoicing — those are Stripe product problems that belong to a different course.
