# Your Working Rhythm

A guide is only as good as the habits you bring to it. This sub-chapter sets three habits that every professional developer works by — and reminds you of them as you go.

---

## Commit every day

At the end of every working session — however small the progress — commit what you have:

```
git add -A
git commit -m "feat(auth): add register endpoint"
git push origin main
```

A daily commit does two things. It creates a checkpoint you can roll back to if something breaks. And it trains the habit that a professional developer has: code that is not committed does not exist.

✅ Before you finish today's work: one clean commit, pushed to GitHub.
❌ Do not go to bed with uncommitted code. You will wake up to a mystery.

---

## Keep secrets out of Git

Your `.env` file was added to `.gitignore` in the project skeleton chapter. But `.gitignore` is a promise, not a lock — `git add .env` overrides it.

Before every commit, glance at `git status`. If `.env` appears, stop and remove it with `git restore --staged .env`. Once a secret is in `git log`, it is permanent — even if you delete the file in the next commit, the secret lives in the history.

✅ Check `git status` before every commit.
❌ Never use `git add .` without looking at what it staged.

---

## Read the error before you ask

When something breaks, do this before searching or asking for help:

1. **Read the full error message** — the stack trace names the file and line. Go there first.
2. **Read the relevant docs** — the official documentation for Express, Mongoose, or Next.js is remarkably clear.
3. **Explain the problem out loud** — describe what you expected to happen, what actually happened, and what you tried. The act of explaining it often reveals the fix.
4. **Ask with evidence** — if you still need help, share: what you expected, what happened, what you tried, and the exact error text. "It doesn't work" is not a question.

✅ Spend 20 minutes debugging before asking. You will find it most of the time.
❌ Do not paste the entire codebase into a chat window. Isolate the problem first.

---

## Use your learning log

After each concept chapter, spend ten minutes writing up what you learned in `learning-log/`. Not a summary — your own explanation of the thing, in your own words. If you can explain it, you understand it. If the words do not come, you have more reading to do.

The learning log is your viva rehearsal. When someone asks you "why JWTs and not sessions?" at a job interview, the answer you wrote three weeks earlier will come back to you.

---

These three habits will serve you long after EduFlow is deployed. Keep them.
