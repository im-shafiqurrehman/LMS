# The Approaches — Sessions vs Tokens

Every authentication system answers one question the same way: HTTP is stateless, meaning the server forgets who you are the moment the response leaves. How does the server remember you between requests?

There are two real answers.

## Sessions (stateful)

The server keeps a record of every active login — a **session** — in a database or in-memory store (Redis is common). When you log in, the server creates a session record and sends back a session ID in a cookie. On every subsequent request, the browser sends that cookie, the server looks up the session ID, and finds your record.

**What's good about it:** revoking a session is trivial — delete the record. Logout is instant and complete. The session can hold arbitrary data (last IP, device, custom flags) and that data can be changed without reissuing anything to the client.

**What's harder about it:** every request triggers a session-store lookup. If you run multiple servers, they all need to reach the same session store — you can't keep sessions in memory locally on each server, because one server won't know about sessions created on another. This is a solvable problem (shared Redis), but it's infrastructure you have to run.

## Tokens (stateless)

The server issues a signed **token** — a self-contained blob carrying the user's ID and role — and sends it to the client. On every subsequent request, the client includes the token. The server verifies the signature (using a secret it holds) and reads the claims directly from the token, with no database lookup.

**What's good about it:** any server in the fleet can verify any token independently. The auth check is CPU-only, not a network round trip. Scaling horizontally is clean.

**What's harder about it:** revoking a token before it expires is genuinely hard. Since the server doesn't keep a record of issued tokens, it can't invalidate one — unless it keeps a blocklist (which reintroduces state). The common mitigation is short expiry: if a token expires in 15 minutes, a leaked one does limited damage.

## The decision for EduFlow

EduFlow uses **JWT-based stateless tokens**, with a two-token pattern to balance the trade-offs:

- An **access token** expires in 15 minutes. It's short-lived precisely because revoking it is hard. If it leaks, the window of exposure is small.
- A **refresh token** expires in 7 days and is stored in the database (hashed). Its only job is to mint a new access token when the old one expires. Because it's stored, it *can* be revoked — logout deletes the stored hash, and any attempt to use the old token fails immediately.

This is the pattern used by most production APIs today. You will build both tokens, understand what each one does, and know why each lives where it does.
