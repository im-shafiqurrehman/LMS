# The Auth Middleware

## Where the project is now

Login issues tokens. No route checks those tokens yet — every route is still open to anonymous requests. This sub-chapter adds the middleware that enforces authentication on protected routes.

## What the middleware does

`authenticate.js` in `src/middleware/` is an Express middleware function. It:

1. Reads the `Authorization` header from the request
2. Expects the format `Bearer <token>` — rejects anything else with 401
3. Calls `jwt.verify(token, JWT_SECRET, { algorithms: ["HS256"] })`
4. If verification fails (expired, tampered, wrong secret) — responds 401
5. If verification succeeds — attaches the decoded payload to `req.user` and calls `next()`

After this middleware runs, any subsequent middleware or route handler can read `req.user.sub` (the user ID) and `req.user.role` without touching the database.

## The shape

```js
// src/middleware/authenticate.js — behaviour spec
// export async function authenticate(req, res, next)
// On failure: res.status(401).json({ error: "Authentication required." })
// On success: req.user = { id: payload.sub, role: payload.role, emailVerified: payload.email_verified }
//             next()
```

✅ Always specify `{ algorithms: ["HS256"] }` in `jwt.verify()` — never allow the library to accept the algorithm from the token header.
❌ Don't attach the full database user object to `req.user` on every request — that's a database query on every authenticated request. The token payload has what you need.

## Add a protected test route

Add a temporary route to verify the middleware works:

```
GET /api/auth/me
```

Protected by `authenticate`. Returns `req.user`. After verifying it works, you can remove it or keep it as a convenience endpoint.

Test:

```bash
# No token — expect 401
curl http://localhost:3000/api/auth/me

# With a valid access token — expect 200 with user payload
curl http://localhost:3000/api/auth/me \
  -H "Authorization: Bearer YOUR_ACCESS_TOKEN"

# With an expired token — expect 401
# (set JWT_ACCESS_EXPIRY=1s temporarily, log in, wait 2 seconds, then try)
```

# Authentication Chapter Checklist

All boxes must be ticked before you continue to Chapter 4 — Authorization.

- [ ] `POST /api/auth/register` creates a user with a hashed password; a duplicate email returns 409
- [ ] The verification OTP is printed to the terminal (or sent by email if you've already wired Resend); it expires after 15 minutes
- [ ] `POST /api/auth/verify-email` marks the account verified; a wrong or expired OTP returns 400
- [ ] `POST /api/auth/login` with correct credentials returns an access token and a refresh token; wrong credentials return 401 with a generic message
- [ ] An unverified account attempting login is rejected with 403
- [ ] `POST /api/auth/refresh` with a valid refresh token returns a new access token; a revoked or expired token returns 401
- [ ] `POST /api/auth/logout` deletes the refresh token; the revoked token cannot be used again
- [ ] `GET /api/auth/me` with a valid access token returns the user payload; without a token it returns 401
- [ ] No readable passwords or refresh tokens exist in the database — only hashes
- [ ] No secret or `.env` value appears in `git log`
