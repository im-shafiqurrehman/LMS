# Authentication Middleware and Chapter Checklist

## The authenticate middleware

Every protected route needs to verify that the incoming request carries a valid, unexpired access token. You write this logic once, in `src/middleware/authenticate.js`, and mount it on any route that requires a logged-in user.

The middleware must:

1. Read the `Authorization` header. The expected format is `Bearer <token>`.
2. If the header is missing or not in that format, return 401.
3. Call `jwt.verify(token, config.jwtSecret)`. If the token is expired or the signature is wrong, `jsonwebtoken` throws — catch it and return 401.
4. Attach the decoded payload to `req.user` — specifically `{ id: payload.sub, role: payload.role }`.
5. Call `next()`.

```js
// src/middleware/authenticate.js — shape only
export function authenticate(req, res, next) {
  const header = req.headers.authorization;
  if (!header?.startsWith('Bearer ')) {
    return res.status(401).json({ error: 'Missing or invalid token' });
  }
  const token = header.split(' ')[1];
  try {
    const payload = jwt.verify(token, config.jwtSecret);
    req.user = { id: payload.sub, role: payload.role };
    next();
  } catch {
    res.status(401).json({ error: 'Token expired or invalid' });
  }
}
```

To protect a route, add the middleware before the handler:

```js
// Example: only authenticated users can access this route
router.get('/me', authenticate, async (req, res) => {
  const user = await User.findById(req.user.id).select('-passwordHash');
  res.json(user);
});
```

✅ A request with a valid token reaches the handler with `req.user` populated.
✅ A request with an expired token receives 401.
✅ A request with no `Authorization` header receives 401.
❌ Do not call `jwt.decode()` (no signature check) instead of `jwt.verify()`. Decode trusts the payload without checking the signature — anyone can forge a payload.

## Add a `GET /api/auth/me` route

This is the simplest protected route: the authenticated user can fetch their own profile. Add it to `auth.routes.js` using the `authenticate` middleware. It should return the user document without the password hash or OTP fields.

## Chapter Definition of Done

- [ ] `POST /api/auth/register` with a new email returns 201 and a success message
- [ ] `POST /api/auth/register` with a duplicate email returns 409
- [ ] `POST /api/auth/register` with an invalid email or short password returns 422
- [ ] `POST /api/auth/verify-email` with the correct OTP marks `emailVerifiedAt` and returns 200
- [ ] `POST /api/auth/login` with correct credentials returns 200 and both tokens
- [ ] `POST /api/auth/login` with an incorrect password returns 401 — same message as a non-existent email
- [ ] `POST /api/auth/refresh` with a valid refresh token returns a new access token
- [ ] `POST /api/auth/logout` deletes the refresh token and returns 200
- [ ] `GET /api/auth/me` without a token returns 401; with a valid token returns the user profile
- [ ] The user's `passwordHash`, `verificationOtp`, and `otpExpiresAt` are never returned in any API response
