# The Register Endpoint

The register endpoint is the first piece of real business logic in EduFlow. It accepts an email and password, creates a user in the database, sends a verification email, and returns a success message — without issuing tokens yet (the account is not verified).

## Where the project is now

You have an empty `auth.routes.js` and `auth.service.js`. The Mongoose `User` model exists. By the end of this sub-chapter, `POST /api/auth/register` is working and a new user document is in the database.

## The validator — reject bad input first

Open `auth.validators.js` and define the register schema using Zod:

```js
// auth.validators.js — shape only
import { z } from 'zod';

export const registerSchema = z.object({
  email: z.string().email('Invalid email address'),
  password: z
    .string()
    .min(8, 'Password must be at least 8 characters')
    .regex(/[A-Z]/, 'Password must contain at least one uppercase letter')
    .regex(/[0-9]/, 'Password must contain at least one number'),
});
```

This runs before any database call. If the request body is missing a field or the email is malformed, the request is rejected with a 422 before the service layer ever sees it.

## The service — the business logic

Open `auth.service.js`. The `register` function must:

1. Call the validator on the request body and throw a 422 if it fails
2. Check whether the email already exists: `User.findOne({ email })`
3. If it does, throw a 409 Conflict error — not a 500, not a vague "error occurred"
4. If it does not, create a new user: `new User({ email, passwordHash: password })` — the `pre('save')` hook will hash it
5. Generate a random 6-digit OTP and set `verificationOtp` and `otpExpiresAt` (15 minutes from now) on the user document
6. Call `user.save()` to persist the document (triggering the password hash hook)
7. Send the OTP to the user's email (stub this for now — wired up in the email chapter)
8. Return `{ message: 'Verification email sent. Please check your inbox.' }`

The service never returns the user's `passwordHash`, `verificationOtp`, or `otpExpiresAt` — those fields are `select: false` in the schema, so they are excluded automatically unless explicitly requested.

## The route

Open `auth.routes.js` and add the register route:

```js
// auth.routes.js — shape only
authRouter.post('/register', async (req, res, next) => {
  try {
    const result = await authService.register(req.body);
    res.status(201).json(result);
  } catch (err) {
    next(err);  // passes to your error-handling middleware
  }
});
```

## Verify it works

Use curl or an API client:

```
POST /api/auth/register
Content-Type: application/json

{
  "email": "alice@example.com",
  "password": "Password123"
}
```

Expected: `201 Created` with `{ "message": "Verification email sent..." }`

Then check MongoDB directly:

```
docker compose exec mongodb mongosh -u admin -p yourpassword eduflow \
  --eval "db.users.findOne({ email: 'alice@example.com' }, { email: 1, role: 1, passwordHash: 1 })"
```

You should see the user document. Notice `passwordHash` starts with `$2b$12$` — that is bcrypt's output format. The plain-text password is gone.

Now try registering again with the same email. You should receive `409 Conflict`.

✅ New user is in the `users` collection; `passwordHash` is a bcrypt hash.
✅ Duplicate email registration returns 409.
✅ Missing or invalid email returns 422 with a message describing the failure.
❌ Do not return the user's passwordHash, OTP, or any sensitive field in the response.
