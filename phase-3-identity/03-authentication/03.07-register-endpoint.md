# The Register Endpoint

## Where the project is now

The auth module exists and is mounted. No routes respond yet. This sub-chapter adds `POST /api/auth/register` — the first working endpoint in EduFlow.

## Request and response contract

The endpoint must accept:

```json
POST /api/auth/register
Content-Type: application/json

{
  "email": "shafiq@example.com",
  "password": "SecurePass123!",
  "role": "student"   // optional — defaults to "student" if omitted; "instructor" allowed; "admin" is rejected
}
```

On success:

```json
HTTP 201 Created
{
  "message": "Account created. Check your email for a verification code."
}
```

On duplicate email:

```json
HTTP 409 Conflict
{
  "error": "An account with this email already exists."
}
```

Notice the success response does **not** return a token. The account exists but is unverified. Tokens are only issued after email verification. This prevents throwaway accounts from getting full access.

## Step 1 — Write the request validator

In `auth.validators.js`, define a Zod schema for the register body. It must:

- Require `email` — valid email format
- Require `password` — minimum 8 characters, at least one uppercase, one lowercase, one digit
- Accept optional `role` — only `"student"` or `"instructor"` allowed; anything else (including `"admin"`) is rejected at the validation layer

Write the schema and export it as `registerSchema`.

**Hint:** Zod's `.refine()` lets you add custom validation logic with a custom error message. You'll need it for the password complexity rules.

## Step 2 — Write the register service function

In `auth.service.js`, implement `registerUser(data)`. It must:

1. Check if a user with that email already exists (Prisma: `findUnique` on `email`)
2. If yes — throw an error with a 409 status code
3. Hash the password with bcrypt at cost factor 12
4. Generate a 6-digit numeric OTP and an expiry 15 minutes from now
5. Create the user record in the database with the hashed password, OTP, and expiry
6. Send a verification email containing the OTP (placeholder for now — the email chapter isn't built yet; `console.log` the OTP to the terminal)
7. Return nothing (the route handler will send the 201)

The OTP generation:

```js
// Shape — generate a 6-digit OTP
// Math.floor(Math.random() * 900000) + 100000  → always 6 digits
// Store it as a string in the database
```

✅ Use `console.log("OTP for", email, ":", otp)` during development — this lets you test verification without a real email service yet.
❌ Don't return the OTP in the API response — that defeats the purpose of email verification.

## Step 3 — Wire the route

In `auth.routes.js`, add:

```js
// POST /api/auth/register
// 1. Validate the body with registerSchema
// 2. Call registerUser(validatedBody)
// 3. Respond 201 with the success message
// Handle errors: 409 for duplicate email, 400 for validation failures
```

Write a small validation middleware that takes a Zod schema, parses `req.body`, and either calls `next()` with the parsed data on `res.locals`, or responds 400 with the Zod error messages. You will reuse this middleware on every route.

## Step 4 — Test it

```bash
curl -X POST http://localhost:3000/api/auth/register \
  -H "Content-Type: application/json" \
  -d '{"email": "test@example.com", "password": "SecurePass1!"}'
```

Expected: `HTTP 201`, the success message in the body, and the OTP printed in your terminal.

Send the same request again:

```bash
# same request
```

Expected: `HTTP 409`, the duplicate email error.

Send a bad password:

```bash
curl -X POST http://localhost:3000/api/auth/register \
  -H "Content-Type: application/json" \
  -d '{"email": "test2@example.com", "password": "weak"}'
```

Expected: `HTTP 400`, a validation error describing why the password was rejected.

Check the database:

```bash
psql $DATABASE_URL -c "SELECT id, email, role, email_verified_at FROM users;"
```

The row should exist with `email_verified_at` as `null`.

✅ All three test cases pass before continuing.
❌ Don't move to the login endpoint until registration works end-to-end — login depends on a registered user existing.
