# What a JWT Actually Is

A **JWT** (JSON Web Token) is three Base64URL-encoded strings joined by dots:

```
header.payload.signature
```

For example:

```
eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9
.eyJzdWIiOiJ1c2VyXzEyMyIsInJvbGUiOiJzdHVkZW50IiwiaWF0IjoxNzAwMDAwMDAwLCJleHAiOjE3MDAwMDA5MDB9
.SflKxwRJSMeKKF2QT4fwpMeJf36POk6yJV_adQssw5c
```

Paste that into [jwt.io](https://jwt.io) and you can read the parts:

**Header** — describes the token type and the signing algorithm:
```json
{
  "alg": "HS256",
  "typ": "JWT"
}
```

**Payload** — the claims. These are the facts the token asserts:
```json
{
  "sub": "user_123",
  "role": "student",
  "iat": 1700000000,
  "exp": 1700000900
}
```

- `sub` (subject) — the user's ID
- `role` — their role in the system
- `iat` (issued at) — Unix timestamp when the token was created
- `exp` (expires at) — Unix timestamp when it stops being valid

**Signature** — computed by hashing `base64(header) + "." + base64(payload)` with your `JWT_SECRET`. This is the only part that proves the token is genuine. If anyone tampers with the payload — changing `"role": "student"` to `"role": "admin"` — the signature no longer matches, and the verification fails.

## What the signature protects — and what it doesn't

The signature proves the token was created by someone who knows the `JWT_SECRET`. It does **not** encrypt the payload. The payload is Base64-encoded, which is reversible by anyone. Do not put sensitive data (passwords, card numbers, PII beyond an ID) in a JWT payload.

What you put in the payload for EduFlow:

```json
{
  "sub": "user_123",
  "role": "student",
  "email_verified": true,
  "iat": 1700000000,
  "exp": 1700000900
}
```

That's it. Keep it small — the token travels in the `Authorization` header of every request.

## Signing and verifying with `jsonwebtoken`

You will use the `jsonwebtoken` npm package. Here is what the API looks like — not the code you write, but the shape so you understand it:

```js
// Signing — takes a payload and the secret, returns a token string
jwt.sign(payload, secret, { expiresIn: "15m" })

// Verifying — takes a token string and the secret, returns the decoded payload
// Throws if the token is expired, tampered with, or signed with a different secret
jwt.verify(token, secret)
```

Install it:

```bash
npm install jsonwebtoken
npm install --save-dev @types/jsonwebtoken
```

> **Interesting to read:** The 2018 "Algorithm Confusion" JWT vulnerability showed that setting `alg: "none"` in the header bypassed signature verification in some libraries. The attack sounds exotic but the lesson is concrete: always explicitly specify `{ algorithms: ["HS256"] }` in your `verify` call — never let the token tell the library which algorithm to use. Search "JWT algorithm confusion attack" if you want the full story.
