# Verify Email, Login, Refresh, and Logout

The register endpoint is done. This sub-chapter adds the four remaining core auth routes: email verification, login, token refresh, and logout.

## Email verification

`POST /api/auth/verify-email` accepts `{ email, otp }`. The service must:

1. Find the user by email: `User.findOne({ email }).select('+verificationOtp +otpExpiresAt')`
   — `select('+field')` is how you fetch a field marked `select: false` when you explicitly need it.
2. If the user does not exist, return 404.
3. If `emailVerifiedAt` is already set, return 409 — the account is already verified.
4. If the OTP does not match or `otpExpiresAt` is in the past, return 422.
5. Set `emailVerifiedAt = new Date()`, clear `verificationOtp` and `otpExpiresAt`, and call `user.save()`.
6. Return `{ message: 'Email verified successfully.' }`

✅ After verification, `emailVerifiedAt` is a timestamp, not null.
✅ A second verification attempt on the same account returns 409.

## Login

`POST /api/auth/login` accepts `{ email, password }`. The service must:

1. Find the user, fetching the `passwordHash` explicitly: `User.findOne({ email }).select('+passwordHash')`
2. If no user is found, return 401 with a generic message: "Invalid email or password." — not "email not found." Leaking which emails are registered is an enumeration vulnerability.
3. Compare the submitted password: `await bcrypt.compare(password, user.passwordHash)`. If it fails, return 401 with the same generic message.
4. If the account is unverified (`emailVerifiedAt` is null), return 403 with "Please verify your email first."
5. Generate a token pair: an access token (JWT signed with `config.jwtSecret`, `sub: user._id, role: user.role`) and a refresh token (a random `crypto.randomBytes(40).toString('hex')`).
6. Hash the refresh token with bcrypt and store it: `new RefreshToken({ tokenHash, userId: user._id, expiresAt })`.
7. Return `{ accessToken, refreshToken, user: { id, email, role } }`.

✅ Login with correct credentials returns `200` and both tokens.
✅ Login with wrong password returns `401` — same message as wrong email.
✅ Login with unverified account returns `403`.

## Token refresh

`POST /api/auth/refresh` accepts `{ refreshToken }`. The service must:

1. Hash the incoming token: `await bcrypt.hash(refreshToken, 10)` — wait, this is wrong. You stored a bcrypt hash, but bcrypt generates a different hash every time. Use `bcrypt.compare()` instead.
   
   **Hint:** You cannot look up a bcrypt hash by re-hashing the token, because bcrypt is non-deterministic. Instead, store an index-friendly fast hash (like `crypto.createHash('sha256').update(token).digest('hex')`) alongside the bcrypt hash, and look up by the fast hash. Or store the raw token hashed with SHA-256 only (acceptable for refresh tokens since they are already random and not passwords). For EduFlow, use SHA-256: `crypto.createHash('sha256').update(token).digest('hex')`.

2. Look up the `RefreshToken` document: `RefreshToken.findOne({ tokenHash })`.
3. If not found or expired, return 401.
4. Look up the user: `User.findById(storedToken.userId)`.
5. Issue a new access token.
6. Return `{ accessToken }`.

## Logout

`POST /api/auth/logout` accepts `{ refreshToken }`. The service must:

1. Hash the incoming token (SHA-256, same as refresh).
2. Delete the record: `await RefreshToken.deleteOne({ tokenHash })`.
3. Return `{ message: 'Logged out.' }` — even if the token was not found (idempotent).

Logout never returns an error. If the token is already gone, the desired state (logged out) is already achieved.

> **Worth reading:** Search "refresh token rotation security" — some systems rotate the refresh token on every use (issue a new one and revoke the old one) to detect token theft. EduFlow uses the simpler fixed-token approach; the rotation pattern is a useful stretch topic.
