# Email Verification

## Where the project is now

Users can register. The OTP is printed to the terminal. The account exists but is unverified ‚Äî `emailVerifiedAt` is `null`. This sub-chapter adds `POST /api/auth/verify-email`.

## Request and response contract

```json
POST /api/auth/verify-email
{
  "email": "test@example.com",
  "otp": "482910"
}
```

On success:
```json
HTTP 200 OK
{ "message": "Email verified. You can now log in." }
```

On wrong OTP:
```json
HTTP 400 Bad Request
{ "error": "Invalid or expired verification code." }
```

## What the service function must do

1. Find the user by email
2. Check that `verificationOtp` matches the provided OTP **and** `otpExpiresAt` is in the future
3. If either check fails ‚Äî return the same error for both (do not distinguish "wrong OTP" from "expired OTP" ‚Äî giving separate errors helps attackers understand which to retry)
4. Set `emailVerifiedAt` to now, and clear `verificationOtp` and `otpExpiresAt`
5. Return success

‚úÖ Always use `bcrypt.compare()` or constant-time comparison for the OTP ‚Äî do not use `===`. Timing attacks on simple string comparison are real.

**Hint:** Prisma's `update` method lets you set multiple fields in one call. You can clear the OTP fields by setting them to `null`.

## Test it

Copy the OTP from your terminal, then:

```bash
curl -X POST http://localhost:3000/api/auth/verify-email \
  -H "Content-Type: application/json" \
  -d '{"email": "test@example.com", "otp": "YOUR_OTP_HERE"}'
```

Check the database ‚Äî `email_verified_at` should now have a timestamp, and the OTP columns should be `null`.

# The Login Endpoint

## Where the project is now

Registration and verification work. Now users can prove who they are and receive tokens.

## Request and response contract

```json
POST /api/auth/login
{
  "email": "test@example.com",
  "password": "SecurePass1!"
}
```

On success:
```json
HTTP 200 OK
{
  "accessToken": "eyJ...",
  "refreshToken": "eyJ..."
}
```

On wrong credentials:
```json
HTTP 401 Unauthorized
{ "error": "Invalid email or password." }
```

Notice the error message is identical for wrong email and wrong password. This is deliberate ‚Äî telling the client *which* was wrong helps attackers enumerate valid accounts.

## What the service function must do

1. Find the user by email ‚Äî if not found, return 401 with the generic error
2. Compare the provided password against `passwordHash` with `bcrypt.compare()`
3. If the password is wrong ‚Äî return 401 with the same generic error
4. Check `emailVerifiedAt` ‚Äî if null, return 403 with "Please verify your email before logging in."
5. Generate an **access token**: sign `{ sub: user.id, role: user.role, email_verified: true }` with `JWT_SECRET`, expiry `JWT_ACCESS_EXPIRY`
6. Generate a **refresh token**: sign `{ sub: user.id }` with `JWT_REFRESH_SECRET`, expiry `JWT_REFRESH_EXPIRY`
7. Hash the refresh token with bcrypt (cost 10 is enough ‚Äî refresh tokens are long random strings, not guessable)
8. Store the hashed refresh token in `refresh_tokens` with `userId` and `expiresAt`
9. Return both tokens

## Token helper

Add a `generateTokenPair(userId, role)` function in `auth.helpers.js` that performs steps 5‚Äì8 and returns `{ accessToken, refreshToken }`. You will call this from login and from the refresh endpoint.

‚úÖ Test: a verified user can log in and receives two tokens. Verify the access token at [jwt.io](https://jwt.io) ‚Äî the payload should show the correct `sub`, `role`, and `exp`.
‚ùå Don't store the access token ‚Äî it is never saved anywhere on the server. Only the refresh token hash is stored.

# The Refresh Endpoint

## Where the project is now

Users have tokens. Access tokens expire in 15 minutes. Without a refresh endpoint, users would be logged out constantly. This sub-chapter adds `POST /api/auth/refresh`.

## Request and response contract

```json
POST /api/auth/refresh
{
  "refreshToken": "eyJ..."
}
```

On success:
```json
HTTP 200 OK
{ "accessToken": "eyJ..." }
```

On invalid or expired token:
```json
HTTP 401 Unauthorized
{ "error": "Invalid or expired refresh token." }
```

## What the service function must do

1. Verify the provided token with `JWT_REFRESH_SECRET` (throws if expired or tampered)
2. Extract `sub` (user ID) from the payload
3. Find all non-expired `RefreshToken` records for that user
4. For each stored hash, run `bcrypt.compare(providedToken, storedHash)` ‚Äî find the matching one
5. If no match found ‚Äî the token was revoked or never issued; return 401
6. Issue a new access token (not a new refresh token ‚Äî the existing one is still valid)
7. Return the new access token

**Hint:** Step 4 is the most expensive part ‚Äî you're running bcrypt comparisons for each stored token. This is why the index on `userId` matters, and why you should delete expired refresh tokens periodically. For now, a simple cleanup on login (delete all expired tokens for the user) is enough.

# Logout

## Where the project is now

You have login and refresh. Logout must revoke the refresh token so it can never be used again.

```json
POST /api/auth/logout
Authorization: Bearer <accessToken>

{
  "refreshToken": "eyJ..."
}
```

On success:
```json
HTTP 200 OK
{ "message": "Logged out." }
```

## What the service function must do

1. Find all refresh token records for the authenticated user (you'll have the user ID from the access token via the auth middleware ‚Äî built in the next sub-chapter)
2. Compare the provided refresh token against each stored hash
3. Delete the matching record
4. Return success

After logout, attempting to use the revoked refresh token at `POST /api/auth/refresh` must return 401. Test this.

‚úÖ Verify that a deleted refresh token cannot be used to get a new access token.
