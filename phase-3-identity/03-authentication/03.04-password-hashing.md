# Why Hashing — and Which Hash

You will never store a password. You will never encrypt a password. You will hash it — and the difference between those three things determines whether a breach is a catastrophe or a recoverable incident.

## Why not store it?

If your database leaks — through an injection, a misconfigured backup, a disgruntled employee — every plaintext password is immediately available to the attacker. And because most people reuse passwords, the attacker now has credentials for Gmail, for their bank, for their employer's VPN. Your breach becomes someone else's breach. This is the single most disqualifying thing you can do as an engineer building anything with user accounts.

## Why not encrypt it?

Encryption is reversible. If you encrypt the password with a key, then the key is also somewhere in your system — and if the attacker has your database, they likely have your config too. A good adversary finds the key and decrypts all the passwords. You've added a step; you haven't solved the problem.

## Why hashing works

A **hash function** is a one-way function: given `password`, it produces a fixed-length fingerprint, and there is no function that goes the other way. When a user logs in, you hash what they typed and compare the result to the stored hash. You never need the original password again.

**The problem with fast hashes (MD5, SHA-1, SHA-256):** they are designed for speed — a modern GPU can compute billions of SHA-256 hashes per second. An attacker with your hash database and a GPU farm can try billions of password guesses per second. At that speed, common passwords fall in seconds; most passwords fall in hours.

**Salting** solves the precomputed lookup problem. A **salt** is a random string generated per password and prepended to the password before hashing. The same password with two different salts produces two completely different hashes, so precomputed tables (rainbow tables) are useless — the attacker has to recompute for each password individually.

**Slow hashes** (bcrypt, scrypt, Argon2) solve the brute-force speed problem. They are deliberately, tunable slow — bcrypt at cost factor 12 takes ~300ms per hash on typical hardware. That is agonising for an attacker trying billions of guesses, and completely imperceptible to a user logging in. Slowness is the feature.

## Which one to use

Use **bcrypt** with a cost factor of 12. It is battle-tested, available in every language, and the default choice of most production systems. Argon2 is the modern winner of the Password Hashing Competition and is strictly better — but bcrypt's track record and library support make it the safer default for a new system. Both are correct choices; bcrypt is simpler to start with.

Install it:

```bash
npm install bcrypt
npm install --save-dev @types/bcrypt
```

The API shape you will use:

```js
// Hashing — takes the plaintext and a cost factor (rounds), returns the hash+salt as one string
bcrypt.hash(plaintext, 12)

// Comparing — takes the plaintext and the stored hash, returns boolean
bcrypt.compare(plaintext, storedHash)
```

bcrypt stores the salt inside the hash string — you don't manage the salt separately. The stored string looks like:

```
$2b$12$LQv3c1yqBWVHxkd0LHAkCOYz6TtxMQJqhN8/LewY9D.Q2Qb8YZuuq
```

The `$12$` is the cost factor — it's stored with the hash so future comparisons use the same settings even if you raise the cost factor later.

✅ Always use `bcrypt.compare()` for comparison — never hash the input and compare strings directly. Timing-safe comparison matters.
❌ Never log the plaintext password anywhere, even "temporarily" during debugging.
