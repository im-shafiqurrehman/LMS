# Authentication — Set the Scene

In 2012, LinkedIn suffered a data breach that exposed 6.5 million password hashes. The hashes were unsalted SHA-1 — a fast, general-purpose hashing algorithm with no per-password randomness. Within days, the majority had been cracked using precomputed lookup tables. Millions of users had their passwords exposed, and because people reuse passwords, the blast radius spread far beyond LinkedIn.

The engineers who built that system probably didn't think hard about hashing. They stored *something* instead of plaintext — that felt like enough. It wasn't. The difference between SHA-1 and bcrypt, between no salt and a per-password salt, is the difference between a breach that destroys user trust and one that is survivable. That difference is about twelve lines of code and one informed decision.

This chapter is that decision, made correctly, from the start.

## What this chapter covers

By the end of this chapter, EduFlow knows who every user is. Specifically:

- A new user can register with an email and password
- The password is never stored in a recoverable form
- After registration, the user receives a verification email; unverified accounts have restricted access
- A verified user can log in and receive an **access token** (short-lived) and a **refresh token** (long-lived)
- The access token proves identity on every subsequent request
- The refresh token can mint a new access token when the old one expires — without requiring re-login
- Refresh tokens can be revoked (logout invalidates them)
- A protected route rejects any request that arrives without a valid, unexpired access token

This is the authentication layer that every other chapter depends on. You build it once, you build it right, and the rest of the system trusts it.
