# The Auth Schema — Your Mongoose Models

Before you write a single route, you need the database models. The Mongoose schema drives everything else — the validation rules, the middleware hooks, the query shape, the token payload. Getting it right now saves you a painful refactor later.

## Where the project is now

You have a running server, a connected Mongoose database, and an empty `src/models/` folder. The `User` collection does not exist yet. This sub-chapter adds the `User` and `RefreshToken` Mongoose models.

## Why Mongoose models — not raw collection inserts

The temptation at this point is to skip defining Mongoose models and write directly to MongoDB collections: `db.collection('users').insertOne({ email, passwordHash })`. This works, but it means:

- No validation — a typo in a field name silently creates a new field in the document
- No hooks — you must manually hash the password in every route handler that writes a password
- No type safety — nothing stops a string where a Date is expected

Mongoose models solve all three. You define the shape once; every write goes through that shape.

## The User model

Create `src/models/user.model.js`. Read what each field does before you write it:

```js
// src/models/user.model.js — shape only
import mongoose from 'mongoose';

const userSchema = new mongoose.Schema(
  {
    email: {
      type: String,
      required: [true, 'Email is required'],
      unique: true,
      lowercase: true,
      trim: true,
    },
    passwordHash: {
      type: String,
      required: true,
      select: false,    // never included in query results by default
    },
    role: {
      type: String,
      enum: ['student', 'instructor', 'admin'],
      default: 'student',
    },
    emailVerifiedAt: {
      type: Date,
      default: null,
    },
    verificationOtp: {
      type: String,
      select: false,
    },
    otpExpiresAt: {
      type: Date,
      select: false,
    },
  },
  { timestamps: true }   // adds createdAt and updatedAt automatically
);
```

Notice:

- **`passwordHash`, not `password`** — the name makes it impossible to accidentally return a plain-text password from a query.
- **`select: false`** on `passwordHash`, `verificationOtp`, and `otpExpiresAt` — these fields are excluded from query results by default. A `User.findOne({ email })` will never include the password hash unless you explicitly say `.select('+passwordHash')`. This prevents accidental exposure in API responses.
- **`emailVerifiedAt: Date, default: null`** — `null` means unverified; a timestamp means verified. This is cleaner than a boolean because it also records _when_ verification happened.
- **`{ timestamps: true }`** — Mongoose adds and maintains `createdAt` and `updatedAt` automatically.

## The password hook

Add this middleware to the user schema, immediately after the field definitions and before the model export:

```js
// Still in user.model.js — add below the schema definition
import bcrypt from 'bcrypt';

userSchema.pre('save', async function (next) {
  // Only hash if the passwordHash field was modified (skips unchanged saves)
  if (!this.isModified('passwordHash')) return next();
  this.passwordHash = await bcrypt.hash(this.passwordHash, 12);
  next();
});
```

This hook runs automatically before every `user.save()`. You never need to remember to hash the password in a route handler — the model handles it. If you later add a password-reset flow, the hashing happens here, not in the reset handler.

✅ Do this: let the `pre('save')` hook own all hashing logic.
❌ Do not hash passwords manually in route handlers. Duplication creates the risk of one path forgetting to hash.

## The RefreshToken model

Create `src/models/refreshToken.model.js`:

```js
// src/models/refreshToken.model.js — shape only
import mongoose from 'mongoose';

const refreshTokenSchema = new mongoose.Schema(
  {
    tokenHash: {
      type: String,
      required: true,
      unique: true,
    },
    userId: {
      type: mongoose.Schema.Types.ObjectId,
      ref: 'User',
      required: true,
    },
    expiresAt: {
      type: Date,
      required: true,
    },
  },
  { timestamps: true }
);

refreshTokenSchema.index({ userId: 1 });
refreshTokenSchema.index({ expiresAt: 1 }, { expireAfterSeconds: 0 }); // TTL index — MongoDB auto-deletes expired tokens

export const RefreshToken = mongoose.model('RefreshToken', refreshTokenSchema);
```

Notice:

- **`tokenHash`, not `token`** — the refresh token value is stored hashed for the same reason passwords are. If the `refreshtokens` collection is ever exposed, nobody gets working tokens.
- **`ref: 'User'`** — this tells Mongoose the `userId` field references the `User` model, enabling `.populate('userId')` if you ever need the user details.
- **`refreshTokenSchema.index({ userId: 1 })`** — you will query by `userId` when revoking all tokens on logout. The index makes that fast.
- **TTL index on `expiresAt`** — `{ expireAfterSeconds: 0 }` tells MongoDB to automatically delete documents when their `expiresAt` timestamp is in the past. Expired refresh tokens clean themselves up without a scheduled job.

## Export both models

Add the export to `user.model.js`:

```js
export const User = mongoose.model('User', userSchema);
```

Then add both models to an `src/models/index.js` barrel file for clean imports across the app:

```js
// src/models/index.js
export { User } from './user.model.js';
export { RefreshToken } from './refreshToken.model.js';
```

✅ Verify: import both models in `server.js` temporarily and log their names (`User.modelName`, `RefreshToken.modelName`). They should print `User` and `RefreshToken`. Remove the debug imports once confirmed.
❌ Do not run `psql` or any SQL commands — this is MongoDB, not a relational database. There is no migration step; the collection is created the first time a document is inserted.

**Hint:** If you see a Mongoose `OverwriteModelError` during hot-reload, it means the model is being registered twice. Guard against it: `mongoose.models.User || mongoose.model('User', userSchema)`.
