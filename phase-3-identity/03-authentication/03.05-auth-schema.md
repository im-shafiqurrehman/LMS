# The Auth Schema ‚Äî Your Database Shape

Before you write a single route, you need the database tables. The schema drives everything else ‚Äî the queries, the validations, the token payload, the checklist. Getting it right now saves you a painful migration later.

## Where the project is now

You have a running server, a connected database, and an empty `schema.prisma`. The `User` table doesn't exist yet. This sub-chapter adds it ‚Äî and the `RefreshToken` table beside it.

## The User model

Open `prisma/schema.prisma` and add the following model. Do not copy-paste ‚Äî read what each field does, then write it yourself:

```prisma
// Shape only ‚Äî write this in your schema.prisma
model User {
  id               String    @id @default(uuid())
  email            String    @unique
  passwordHash     String
  role             Role      @default(STUDENT)
  emailVerifiedAt  DateTime?
  verificationOtp  String?
  otpExpiresAt     DateTime?
  createdAt        DateTime  @default(now())
  updatedAt        DateTime  @updatedAt

  refreshTokens    RefreshToken[]
  // (more relations added in later chapters)
}

enum Role {
  STUDENT
  INSTRUCTOR
  ADMIN
}
```

Notice:
- `passwordHash` ‚Äî not `password`. The name makes it impossible to accidentally return a plaintext password from a query.
- `emailVerifiedAt` ‚Äî nullable. `null` means unverified; a timestamp means verified. This is cleaner than a boolean because it also records *when* verification happened.
- `verificationOtp` and `otpExpiresAt` ‚Äî the one-time code sent to the email, and when it expires. You'll use these in the email verification sub-chapter.

## The RefreshToken model

```prisma
model RefreshToken {
  id        String   @id @default(uuid())
  tokenHash String   @unique
  userId    String
  user      User     @relation(fields: [userId], references: [id], onDelete: Cascade)
  expiresAt DateTime
  createdAt DateTime @default(now())

  @@index([userId])
}
```

Notice:
- `tokenHash` ‚Äî not `token`. The refresh token value is stored hashed for the same reason passwords are. If the `refresh_tokens` table leaks, nobody gets working tokens.
- `onDelete: Cascade` ‚Äî when a user is deleted, their refresh tokens are deleted too. No orphaned rows.
- `@@index([userId])` ‚Äî you will query by `userId` when revoking all tokens on logout. The index makes that fast.

## Run the migration

```bash
npx prisma migrate dev --name add-user-and-refresh-token
```

Verify in your database:

```bash
psql $DATABASE_URL -c "\dt"
```

You should see `users` and `refresh_tokens` in the table list.

‚úÖ Check: `npx prisma studio` opens the visual browser and shows both tables with the correct columns.
‚ùå Don't manually edit the database schema outside of Prisma migrations ‚Äî your `schema.prisma` and the real database will drift out of sync, and the mismatch will surface at the worst possible time.

**Hint:** If the migration fails with "column already exists," you may have run a partial migration earlier. Check `prisma/migrations/` for incomplete migration folders and remove them before retrying.
