# The Me Endpoint and Route Summary

The GET /api/auth/me route is the simplest protected route. It proves the authentication middleware works end-to-end.

Test it:

```
# 1. Register
curl -X POST http://localhost:3000/api/auth/register \
  -H "Content-Type: application/json" \
  -d '{"email":"test@example.com","password":"Test1234!"}'

# 2. Verify email (check server console for OTP)
curl -X POST http://localhost:3000/api/auth/verify-email \
  -H "Content-Type: application/json" \
  -d '{"email":"test@example.com","otp":"123456"}'

# 3. Login
curl -X POST http://localhost:3000/api/auth/login \
  -H "Content-Type: application/json" \
  -d '{"email":"test@example.com","password":"Test1234!"}'
# Copy the accessToken from the response

# 4. Me
curl -X GET http://localhost:3000/api/auth/me \
  -H "Authorization: Bearer YOUR_ACCESS_TOKEN_HERE"
```

Expected: { "id": "...", "email": "test@example.com", "role": "student", ... }

**Complete auth route table:**

| Method | Path | Middleware | Handler |
|--------|------|-----------|---------|
| POST | /api/auth/register | validate(registerSchema) | register |
| POST | /api/auth/verify-email | validate(verifyEmailSchema) | verifyEmail |
| POST | /api/auth/login | validate(loginSchema) | login |
| POST | /api/auth/refresh | validate(refreshSchema) | refresh |
| POST | /api/auth/logout | validate(logoutSchema) | logout |
| GET | /api/auth/me | authenticate | me |
