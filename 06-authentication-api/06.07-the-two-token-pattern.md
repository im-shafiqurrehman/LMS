# The Two-Token Pattern

A single JWT with a long expiry solves one problem and creates another. A 7-day JWT is convenient — the user stays logged in. But if that token is stolen, the attacker has 7 days to use it, and you have no way to stop them without invalidating everyone's sessions.

The solution is two tokens with different roles:

**Access token** — short-lived (15 minutes by default). The client includes this in the Authorization: Bearer header on every API request. When it expires, the client does not ask the user to log in again — it uses the refresh token to get a new one silently.

**Refresh token** — long-lived (7 days). The client stores this and sends it only to POST /api/auth/refresh. On the server, the refresh token is stored hashed in the database. To revoke a session (logout), you delete the refresh token record. The access token may still be valid for up to 15 minutes, but it cannot be renewed — the session is effectively ended.

```
Login:
  Server issues:  accessToken (15m JWT)  +  refreshToken (random 40-byte string)
  Server stores:  SHA-256 hash of refreshToken in RefreshToken collection
  Client stores:  accessToken in memory  +  refreshToken in HttpOnly cookie (or secure storage)

On every request:
  Client sends:   Authorization: Bearer <accessToken>
  Server:         jwt.verify(accessToken, JWT_SECRET) → req.user

Access token expires:
  Client sends:   { refreshToken } to POST /api/auth/refresh
  Server:         finds RefreshToken by SHA-256 hash → issues new accessToken

Logout:
  Client sends:   { refreshToken } to POST /api/auth/logout
  Server:         RefreshToken.deleteOne({ tokenHash }) → session revoked
```

> Worth reading: search "JWT refresh token rotation security" for the tradeoffs of rotating refresh tokens on every use.
