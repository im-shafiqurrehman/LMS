# The Approaches — How Authentication Is Done

Before you write a line of auth code, you need to understand the two broad ways a web API can prove who a user is on each request — and why this project uses one and not the other.

## Sessions vs tokens

**Session-based authentication** is the older approach. On login, the server creates a session record in the database (or in memory), generates a session ID, and sends that ID to the browser as a cookie. On every subsequent request, the browser sends the cookie; the server looks up the session ID in the database to find out who the user is.

This works well for traditional server-rendered applications. The problem for EduFlow is that it introduces **statefulness** — the server must store and look up session records. In a horizontally scaled system (multiple server instances), every instance needs access to the same session store. That means shared state between servers, which adds complexity and a potential single point of failure. It also makes the mobile API more awkward: cookies are a browser concept and require extra setup on native clients.

**Token-based authentication (JWT)** takes a different approach. On login, the server creates a **signed token** that contains the user's identity and role, and sends it to the client. On every subsequent request, the client sends the token in an `Authorization` header; the server **verifies the signature** rather than looking anything up in a database. The server is stateless — it does not need to remember anything between requests.

For EduFlow, this is the right approach. The frontend is a Next.js app and the plan includes a mobile API; JWTs work cleanly for both. The server is stateless, which makes horizontal scaling straightforward.

The trade-off: **you cannot instantly invalidate a JWT** the way you can delete a session. If a user's token is stolen, there is no central record to delete. You handle this with short expiry times on the access token (15 minutes), and a separately revocable refresh token stored in the database.

## The two-token pattern

A single JWT with a long expiry solves one problem and creates another. A 7-day JWT is convenient — the user stays logged in. But if that token is stolen, the attacker has 7 days to use it, and you have no way to stop them.

The solution is to split authentication into two tokens:

- **Access token** — short-lived (15 minutes). The client includes this on every API request. When it expires, the user is not asked to log in again; the client uses the refresh token to get a new one silently.
- **Refresh token** — long-lived (7 days). The client stores this securely and sends it only to the `/auth/refresh` endpoint. On the server, the refresh token is stored hashed in the database. To revoke a session (logout), you simply delete the refresh token record. The access token may still be valid for up to 15 minutes, but it cannot be renewed — the session is effectively dead.

This is the pattern you will implement. It gives you long user sessions without giving a stolen token indefinite access.
