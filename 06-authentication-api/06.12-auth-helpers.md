# Auth Helpers — Token Generation and OTP

Create server/src/modules/auth/auth.helpers.js. These small utilities keep the service layer clean.

```js
// src/modules/auth/auth.helpers.js — shape only
import jwt    from 'jsonwebtoken';
import crypto from 'crypto';
import { config } from '../../config/index.js';

// Issue a short-lived access token
export function issueAccessToken(userId, role) {
  return jwt.sign(
    { sub: userId.toString(), role },
    config.jwtSecret,
    { expiresIn: config.jwtAccessExpiry }  // "15m"
  );
}

// Issue a long-lived refresh token (random bytes, not JWT)
export function issueRefreshToken() {
  return crypto.randomBytes(40).toString('hex');
}

// Hash a refresh token for storage (SHA-256 — fast, for lookup)
export function hashRefreshToken(token) {
  return crypto.createHash('sha256').update(token).digest('hex');
}

// Generate a 6-digit OTP and its expiry
export function generateOtp() {
  const otp       = Math.floor(100000 + Math.random() * 900000).toString();
  const expiresAt = new Date(Date.now() + 15 * 60 * 1000); // 15 minutes
  return { otp, expiresAt };
}
```

**Why SHA-256 for the refresh token hash, not bcrypt?**

bcrypt is non-deterministic — the same input produces a different hash every time (because the salt is random). You cannot look up a bcrypt hash in the database by re-hashing the token. SHA-256 is deterministic: the same token always produces the same hash. Combined with the fact that refresh tokens are already long random strings (not user-chosen passwords), SHA-256 is sufficient.
