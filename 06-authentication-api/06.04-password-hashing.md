# Password Hashing

Storing passwords is a trap that has caught many engineering teams. Understanding why the naive approach fails — and exactly what bcrypt does differently — is what this sub-chapter covers.

## The wrong approaches

**Plain text.** Never store a password as the user typed it. If the database is ever read by an attacker (a breach, a misconfigured backup, an exposed admin panel), every user's password is immediately exposed. Users reuse passwords; a single leaked plaintext list can compromise their accounts on dozens of other services.

**Fast hashing (MD5, SHA-256).** Hashing the password before storing it feels like enough. It is not. Fast hash functions are designed to be computed billions of times per second on modern hardware. An attacker with a copy of your hashed passwords can run a dictionary attack — hashing millions of common passwords and comparing — and crack most weak passwords in minutes. The speed that makes SHA-256 great for checksums makes it dangerous for passwords.

**Fast hashing with a global salt.** Adding a fixed string to every password before hashing is a marginal improvement. If the salt is discovered, the attacker can still run a dictionary attack with the known salt.

## The right approach: bcrypt with per-password salts

**bcrypt** was designed specifically for password hashing. It is deliberately slow — the cost factor you pass in controls how many iterations of the hash are computed. The current recommended cost is `12`. At cost 12, a single bcrypt computation takes approximately 250ms on modern hardware. That 250ms is acceptable for a login request; it is crippling for an attacker trying to brute-force millions of passwords.

bcrypt also generates a unique, random salt for every password and embeds it in the output hash. Two users with identical passwords produce different hashes. Precomputed rainbow tables are useless because the salt is unique per password.

```js
import bcrypt from 'bcrypt';

// Hash a password (at registration)
const passwordHash = await bcrypt.hash(plainTextPassword, 12);
// Store passwordHash in the database — never the plain text

// Verify a password (at login)
const isMatch = await bcrypt.compare(plainTextPassword, storedHash);
// true if the password matches; false if not
```

Install bcrypt:

```
npm install bcrypt
npm install --save-dev @types/bcrypt
```

## Where hashing happens in EduFlow

You will hash passwords in two places: in the Mongoose `pre('save')` middleware on the User model (so every save that modifies the password field hashes it automatically), and in the password-reset endpoint when a user changes their password. You never store the plain-text password anywhere — not in a log, not in a temporary variable in a route handler, not in a debug print statement.
