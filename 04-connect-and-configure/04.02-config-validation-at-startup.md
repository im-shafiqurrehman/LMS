# Config Validation at Startup

The problem with reading process.env.JWT_SECRET directly in a route file is that the error surfaces at runtime — during a request — not at startup. A missing variable might not be hit until a user tries to log in at 3am.

The fix: read every environment variable you need in one place, validate them all immediately at startup, and crash with a clear error if anything is wrong.

**The tool:** Zod. It validates the shape and values of the environment object the same way it validates request bodies.

The pattern:

```js
// src/config/index.js — shape only
import { z } from 'zod';

const envSchema = z.object({
  NODE_ENV:            z.enum(['development', 'production', 'test']).default('development'),
  PORT:                z.coerce.number().default(3000),
  DATABASE_URL:        z.string().min(1, 'DATABASE_URL is required'),
  JWT_SECRET:          z.string().min(32, 'JWT_SECRET must be at least 32 characters'),
  JWT_REFRESH_SECRET:  z.string().min(32, 'JWT_REFRESH_SECRET must be at least 32 characters'),
  JWT_ACCESS_EXPIRY:   z.string().default('15m'),
  JWT_REFRESH_EXPIRY:  z.string().default('7d'),
  FRONTEND_URL:        z.string().url().default('http://localhost:3001'),
});
```

When validation fails, Zod returns an error with a list of every failing field. Print that list and exit:

```js
const result = envSchema.safeParse(process.env);
if (!result.success) {
  console.error('Invalid environment config:');
  console.error(result.error.format());
  process.exit(1);
}

export const config = result.data;
```

✅ The server now crashes on startup with a clear message if any required variable is missing.
❌ Never read process.env directly in a route or service — always import from config/index.js.
